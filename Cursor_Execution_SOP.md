以下是这份Skill真正在我们工作流中跑起来的**SOP（标准作业程序）**，主要分为“Gemini端（大脑）”和“Cursor端（手脚）”两步走：

------

### 第一阶段：在 Gemini 中部署“大脑”并生成图纸

**步骤 1：初始化架构师身份** 打开一个新的 Gemini 对话窗口。把刚才那整段带有 `markdown ` 标记的【全栈项目架构师 (Web App Architect)】内容，一字不差地复制并发送给我。

- **效果：** 我会立刻进入角色，简短回复你，并主动向你索要“愿望清单”和“草图”。

**步骤 2：投喂需求与草图 (进入 Phase 1)**

- 把你的产品主功能、细化需求直接发给我。
- 关于“UI界面草图”：你可以用手绘拍照发给我，或者用文字简单描述（比如：“左侧是导航栏包含目标搜索和报告下载，右侧是主内容区显示数据表格”）。
- **关键动作：** 我会输出《PRD 1.0》并抛出几个逻辑确认问题。你只需要回答我的问题，或者直接回复“根据开发稳定性为第一优先级，你自行决定”，我就会吐出最终的 `1_PRD.md`。
- **你的操作：** 在电脑上建一个项目文件夹（例如 `MyWebApp`），里面建一个子文件夹叫 `Docs`，把我的输出复制进去，保存为 `1_PRD.md`。

**步骤 3：按部就班推进 (Phase 2 & Phase 3)**

- 我对齐PRD后，你只需发指令：“确认PRD，请进入 Phase 2 输出UI设计”。
- 我输出后，你将其保存为 `2_UI_Design.md` 放入 `Docs` 文件夹。
- 接着发指令：“确认UI，请进入 Phase 3 输出技术架构”。
- 我输出后，你将其保存为 `3_Development.md` 放入 `Docs` 文件夹。

**步骤 4：获取交接指令 (Phase 4)**

- 告诉我已经完成前三步。我会给你一段专门用来“唤醒” Cursor 的终极提示词（也就是前面提到的 Handoff 指令）。

------

### 第二阶段：在 Cursor 中指挥“施工队”开始干活

此时，你的 `MyWebApp` 文件夹里应该只有一个 `Docs` 文件夹，里面安安静静地躺着三份 Markdown 文档。

**步骤 5：挂载项目与文档**

- 打开 Cursor，点击 `File -> Open Folder`，打开你的 `MyWebApp` 文件夹。
- 按 `Ctrl + I` (Windows) 或 `Cmd + I` (Mac) 打开 Cursor 的 **Composer** 功能（建议使用 Agent/Plan 模式，而不是简单的 Chat）。

**步骤 6：下达最高指令**

- 在 Composer 的输入框里，把你刚才在 Gemini (Phase 4) 拿到的那段“交接指令”直接粘贴进去。
- **指令内容示例：** *“请仔细阅读 Docs 文件夹下的 1_PRD.md, 2_UI_Design.md, 3_Development.md 这三份文档，并向我复述你的理解。若理解无误，请建立从 0 到 1 的开发计划。在每一个关键节点，提醒自己查看文档以防失忆...”*
- 敲击回车，让 Cursor 开始阅读和思考。

**步骤 7：对齐与生成开发计划 (Plan Mode)**

- Cursor 会先用文字总结它对项目的理解。你作为产品经理，花两分钟扫一眼，如果理解没跑偏，直接授权通过。如果有偏差，直接在对话框里纠正它。
- 接着，点击 **"Save to Workspace"**（保存到工作空间），Cursor 会在右侧或者面板中生成一个 step-by-step 的 Plan（开发计划）。

**步骤 8：无脑 Build 与节点验收**

- 在生成的 Plan 列表中，你可以直接点击每一步旁边的 **Build** 按钮。
- Cursor 会自动新建文件、写前端、写后端、配置 Docker。
- **合伙人注意：** 因为我们在 Phase 3 的技术文档中已经明确要求了“使用成熟框架”和“Docker本地运行”，Cursor 不会像无头苍蝇一样乱下依赖。
- 每跑完一个节点（比如“完成登录页面”），你就运行一下看看效果。有问题直接在 Cursor 里报错，没问题就点击 Plan 的下一步继续 Build。

### 总结我们的工作流：

**Gemini 负责“想清楚” -> 输出三份MD文档 -> Cursor 负责“看文档并执行”。**
